MCP Auth: OAuth 2.0 + Dynamic Client Registration

MCP is Lucid's natural language API — you send a plain-English prompt like "find my most recent document" and the MCP server translates it into the right Lucid API calls and returns the result. To do that on your behalf it needs an access token, which requires OAuth. And unlike the REST API, MCP has no pre-registered client ID — so the server registers itself on the fly. That self-registration is DCR.

Why does MCP use DCR instead of a fixed client ID? MCP is a protocol designed for open, dynamic ecosystems — any server should be able to connect without requiring every developer to register in advance. There's no central portal where every MCP client gets listed. DCR lets each server introduce itself at runtime and receive credentials on the spot, so the ecosystem scales without manual overhead.

This App (IdP)

              

            


            
            


              

                

                  Step 1
                  User visits Lucid
                

                

                  
The user navigates to lucid.app. Lucid detects no active session for your domain and looks up the SAML configuration for your account.

                  
Browser → Lucid SP

                

              


              

                

                  Step 2
                  Lucid builds AuthnRequest → redirects to IdP
                

                

                  
Lucid (acting as SP) generates a signed AuthnRequest XML, base64-encodes and deflate-compresses it, and redirects the browser to your IdP SSO URL with it as a query parameter.

                  
Lucid SP → Browser → IdP SSO URL

                  

                    
                    
<samlp:AuthnRequest
  xmlns:samlp="urn:oasis:names:tc:SAML:2.0:protocol"
  ID="_abc123..."
  Version="2.0"
  IssueInstant="2026-02-26T12:00:00Z"
  Destination="http://localhost:8000/saml/sso"
  AssertionConsumerServiceURL="https://lucid.app/saml/sso/yourdomain?idpHash=..."
  ProtocolBinding="urn:oasis:names:tc:SAML:2.0:bindings:HTTP-POST">
  <saml:Issuer>https://lucid.app</saml:Issuer>
</samlp:AuthnRequest>

                  

                

              


              

                

                  Step 3
                  IdP authenticates user
                

                

                  
In a real IdP (Okta, Entra) this is where the login form appears. Here, the user is pre-configured — we assemble their attributes from the Setup panel and build the assertion.

                  

                    
                  

                

              


              

                

                  Step 4
                  IdP signs assertion with RSA-SHA256
                

                

                  
The Assertion XML is signed using the private key — Lucid holds the matching public certificate and uses it to verify the signature. Any tampering after signing invalidates the assertion.

                  

                    
                    
Run a Dry Run first to see the XML here.

                  

                  

                

              


              

                

                  Step 5
                  Browser POSTs SAMLResponse to Lucid ACS
                

                

                  
The IdP renders a hidden HTML form containing the base64-encoded SAMLResponse. The browser auto-submits it via HTTP POST directly to Lucid's ACS URL — the IdP never touches Lucid's servers directly.

                  

                    
                    
<form method="POST" action="https://lucid.app/saml/sso/...">
  <input type="hidden" name="SAMLResponse" value="PHNhbWxwOlJlc3BvbnNlI..." />
  <input type="hidden" name="RelayState" value="..." />
</form>
<script>document.forms[0].submit();</script>

                  

                  

                

              


              

                

                  Step 6
                  Lucid validates assertion → session established
                

                

                  
Lucid's ACS handler verifies: the signature (using the registered cert), the Issuer (must match your configured IdP entity ID), the Audience (must be Lucid's entity ID), and NotOnOrAfter (must be in the future). If all pass, Lucid creates a session and redirects the user in.

                  

                    

                      ◌
                      Signature verified against registered certificate
                    

                    

                      ◌
                      Issuer matches configured IdP Entity ID
                    

                    

                      ◌
                      Audience matches Lucid's SP Entity ID
                    

                    

                      ◌
                      NotOnOrAfter is in the future (not expired)
                    

                    

                      ◌
                      User.email attribute present and maps to a Lucid account
